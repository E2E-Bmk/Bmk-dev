## 1. Introduction

Large language models have demonstrated remarkable progress in code generation, achieving strong performance on function-level benchmarks such as HumanEval and MBPP, and even tackling repository-level tasks like bug fixing in SWE-bench. These advances have fueled a wave of AI-powered coding assistants that are increasingly deployed in real-world software development.

However, a growing body of evidence suggests that headline pass rates on popular coding benchmarks may not reflect genuine capability gains. Audits of SWE-bench Verified revealed fundamental design flaws and training contamination, leading frontier models to achieve high scores without corresponding improvements in real-world software engineering. More recently, a detailed audit of SWE-Bench Pro — previously recommended as a successor — found that approximately 30% of tasks are broken, with failures driven by four recurring validity issues: tests that enforce implementation details not stated in the prompt, prompts that omit requirements hidden tests require, tests with insufficient coverage that allow incomplete fixes to pass, and prompts that mislead models toward incorrect behavior. The root cause is structural: these benchmarks are programmatically derived from pull-request histories, where tests are written to validate a specific change rather than to define an implementation-agnostic behavioral standard. As frontier models improved from 23% to 80% pass rate on SWE-Bench Pro within eight months, it became increasingly unclear whether rising scores measured better agents or simply noisier evaluation.

**Separating signal from noise in coding evaluations has become an urgent prerequisite for any meaningful benchmark.** Yet even a trustworthy pass rate answers only whether a model succeeded on a task — not whether the task measures the capability we care about.

The core difficulty of real-world software engineering lies not in producing locally correct code fragments, but in maintaining **system-wide consistency** — ensuring that independently authored modules share coherent state representations, honor interface contracts, and propagate errors correctly across component boundaries. This cross-module integration capability is what separates a collection of correct functions from a functioning system. **Do current LLMs possess this capability?** Despite the proliferation of code generation benchmarks, this question remains unanswered — not because it is unimportant, but because existing evaluation frameworks are structurally unable to isolate it.

Current benchmarks fall into two categories, neither of which isolates system integration. Function-level benchmarks (HumanEval, MBPP, LiveCodeBench) evaluate local correctness on isolated problems. Repository-level benchmarks take two forms: *modification-based* benchmarks (SWE-bench, Aider-bench) test the ability to patch existing codebases, where the surrounding architecture is already provided; *generation-based* benchmarks (NL2RepoBench, PRDBench, ProgramBench) ask models to produce entire repositories, but primarily measure **specification-following ability** — whether a model can faithfully implement a detailed blueprint. NL2RepoBench, for instance, provides specifications exceeding 6,000 words per task, including complete API signatures, usage examples, and edge-case descriptions. This design is well-suited for evaluating implementation fidelity, but it largely eliminates the need for autonomous architectural decisions and cross-module reasoning — the specification already prescribes how modules should be organized and how they interact. As a result, a complementary but distinct capability remains unmeasured: **system design ability**, i.e., whether a model can independently make consistent architectural decisions given only behavioral constraints.

To address both the validity crisis and the construct gap, we introduce **Spec2Repo**, a benchmark designed with two layered objectives:

**Layer 1 — Trustworthy evaluation.** Unlike PR-derived benchmarks, Spec2Repo tasks are constructed forward from behavioral specifications written by experienced developers, not backward from merged patches. Our design directly targets the four validity failure modes identified in recent audits: (i) behavioral specifications prevent training-evaluation leakage by excluding implementation blueprints and code examples; (ii) test suites are rewritten to verify *behavior* rather than *implementation interfaces*, avoiding overly strict tests that invalidate functionally correct solutions; (iii) oracles are derived from original repository test suites with systematic filtering, ensuring coverage beyond what the specification explicitly states while removing contradictions; (iv) specification calibration follows explicit inclusion/exclusion criteria, reducing underspecification and misleading prompts. This forward construction paradigm replaces the PR-derived approach that OpenAI and others have shown to be systematically unreliable.

**Layer 2 — Isolating system integration.** Even with trustworthy evaluation, a benchmark must measure a capability that existing suites do not. Spec2Repo isolates cross-module system integration by providing only behavioral constraints (~1,000 words) and a minimal test-interface contract, while oracles test behaviors the specification does not fully prescribe. Models must autonomously decide internal architecture, module boundaries, and cross-module data flow. We employ a two-layer evaluation framework — unit tests versus integration tests — and define the **Integration Gap** (unit pass rate minus integration pass rate) as a diagnostic metric. A large gap indicates that the model produces individually correct components but fails to assemble them into a coherent system.

Our main contributions are as follows:

- **High-fidelity benchmark construction.** We present Spec2Repo, a benchmark of [N] end-to-end Python project generation tasks built through a forward specification-and-oracle pipeline that systematically avoids the validity failure modes prevalent in PR-derived coding benchmarks. We validate task quality through [broken-rate analysis / oracle verification on gold code].

- **Integration capability isolation.** To our knowledge, this is the first benchmark that decouples cross-module system integration from specification-translation ability. We evaluate [M] frontier LLMs and find a substantial Integration Gap: models achieve [X]% unit test pass rate but only [Y]% integration test pass rate.

- **Failure mode analysis.** Manual classification of 100 integration failures reveals that [Z]% stem from cross-module consistency issues (state inconsistency, interface contract violations, error propagation breakage) rather than single-module logic errors, confirming that the gap measures a distinct capability deficit.

- **Specification-Integration tradeoff.** Through controlled ablation on NL2RepoBench tasks, we characterize how Integration Gap varies with specification granularity, revealing the relationship between specification detail and the model's reliance on autonomous system design.

- **Human baseline.** Experienced developers achieve an Integration Gap of less than [H]%, confirming that the observed gap reflects a model-specific deficiency rather than inherent task difficulty.

Our findings have direct implications for both evaluation and deployment. On the evaluation side, they demonstrate that benchmark trustworthiness and construct validity are orthogonal concerns — a benchmark can be free of broken tasks yet still fail to measure the capabilities that matter. On the deployment side, they identify cross-module consistency as a concrete gap that current models cannot reliably bridge, delineating where AI-assisted development can autonomously generate components and where human oversight remains essential for system-level integration.
