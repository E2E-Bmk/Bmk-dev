研究提案

Spec2Repo

首次量化 LLM 仓库级代码生成中的集成能力缺口

当前的 AI 编程助手能写出正确的组件，但拼不成正确的系统。 Spec2Repo 是第一个能够将组件正确性与集成正确性分开量化的代码生成基准。 通过行为规约驱动的任务设计和分层行为测试， 我们发现即使最强模型的 Integration Gap 也高达 [Z] 个百分点—— 而人类开发者在同一任务上的 Gap 不到 [H]%。

核心问题
组件写对了，系统却跑不了

随着代码生成从函数级（HumanEval）走向仓库级（NL2Repo、RepoZero）， 真实的能力瓶颈正在浮现：模型生成的单个组件各自能通过自测， 但组装成多组件系统后频繁出现行为错误—— 接口类型不匹配、状态表示不一致、错误在模块边界处被静默吞掉。

这意味着仓库级代码生成实际包含两种可分离的能力： 组件正确性（单个模块的输入输出是否正确） 和 集成正确性（模块之间能否正确交互）。 这两种能力对应不同的认知需求和改进方向—— 组件错误需要更好的代码生成，集成错误需要更好的架构推理。

关键洞察

现有基准在评估框架层面缺乏区分这两种能力的机制—— 既无法确认这一现象是否系统性存在， 也无法指导 coding agent 的能力发展方向。 评估从诊断工具退化成了排行榜数字。

结构性障碍
为什么现有基准做不到：不是加几个测试的事

这不是前人疏忽，是任务设计的结构性问题。 函数级基准根本没有集成维度；补丁级基准在已有架构上做局部修改，集成结构是给定的。 新兴的仓库级基准本应是检验集成能力的理想场所，但均存在结构性障碍：

基准模型输入实际测试内容为什么无法隔离集成能力
HumanEval / MBPP 函数签名 + 文档字符串 单函数正确性 无集成维度
SWE-bench 完整代码库 + Issue 补丁应用 架构已给定  模型只做局部修改，不需要自主集成决策
NL2RepoBench 自然语言规约（~19K tokens） 按规约逐文件实现 Spec-following  规约已包含完整模块结构和 API 设计，集成退化为翻译蓝图
RepoZero 显式 API 规约 + 跨语言 跨语言重现 Spec-following  架构被 API 签名预先限定
ProgramBench 编译后的二进制 + 文档 行为等价重建 落点偏移  逆向设计使其主要考察模型的信息收集能力，而非架构决策；且无规约支撑分层测试，只能端到端评估
Spec2Repo ✦ 仅行为规约（~1K 词） 自主架构 + 分层评估 同时满足两个条件  行为规约迫使自主决策 + 分层行为测试解耦得分

共同缺口：要隔离集成能力，行为规约和分层行为测试缺一不可—— 前者迫使模型自主做架构决策并为 implementation-agnostic 评估提供依据， 后者解耦两种能力的得分， 同时基于行为而非实现细节的测试避免将命名差异误判为逻辑错误。

条件一 — 行为规约

迫使自主架构决策

规约只描述可观测行为——输入输出契约、错误处理预期。 不泄露模块划分、API 签名或目录布局。 模型必须自己决定内部结构。 同时为 implementation-agnostic 的分层测试提供评估依据。

条件二 — 分层行为测试

解耦两种能力的得分

Unit test 验证单组件输出正确性，Integration test 验证组件交互正确性。 测试基于行为而非实现细节， 任何功能正确的实现都能通过，不限命名、类层次或文件组织。

Spec2Repo 设计
行为规约 × 分层评估 = Integration Gap

Spec2Repo 同时满足上述两个条件。每个任务提供 ~1000 词的行为规约， 模型从空目录出发，自主决定内部架构、模块边界和跨模块数据流， 构建完整可运行项目。 评估通过分层 oracle 测试独立打分，定义 Integration Gap = Unit Pass Rate − Integration Pass Rate， 直接量化集成能力缺口。

Oracle 约束一 — 行为性

任何正确的重实现都能通过，无论内部结构如何

测试仅检查可观察输出和副作用。 对 repr 格式、内部字段名或异常消息文本的断言被排除—— 它们反映的是某个具体实现，而非行为契约。

构建流水线
任务如何被构造

每个 Spec2Repo 任务通过五阶段流水线从真实开源 Python 库构建。 流水线结构确定但执行迭代：后续阶段可向前反馈， 任务仅在所有质量检查通过后方可毕业。

  1. 候选库筛选。 Python 库需通过硬门槛：活跃测试套件、 公开文档足以撰写行为规约、 无二进制或大数据文件依赖。
  2. 行为规约撰写。 从公共 API 面撰写行为规约—— 库对调用者的承诺、错误抛出条件、跨接口不变量。 规约读起来像库文档，不像基准测试。 每个纳入决策受两个问题约束： 这是否属于公共行为契约？是否不可仅凭领域知识推导？
  3. Oracle 测试过滤。 对原有测试套件进行审计，排除检查内部状态的测试 （repr 格式、私有字段名、精确异常文本）。 覆盖不足处从规约生成补充测试， 每个生成的测试在 spec_test_map 中逐行关联到规约章节。 按 unit / integration 分层标注。
  4. 候选评估。 候选模型仅收到行为规约，从空目录产出完整实现。 在三个粒度上评分：unit、integration、system end-to-end。 导入来源检查确认模型包来自候选目录。
  5. 任务裁决。 裁判在读取分数之前先审计评分运行的作弊、可解性和公平性。 失败分两轮分类： 先检查工具有效性（测试是否检查了文档化的可观察行为）， 再检查模型能力（失败是否可追溯到候选输出中的真实能力差距）。
预期贡献
Spec2Repo 为社区带来什么
  • 1
    Integration Gap 作为诊断性 construct。 首次将组件正确性和集成正确性解耦为可独立量化的指标， 揭示现有评估遮蔽的能力维度。 这一指标不仅量化差距，更指明改进方向—— 跨模块架构推理，而非更好的单组件代码生成。
  • 2
    同时满足两个必要条件的基准设计。 行为规约迫使模型自主做架构决策， 分层行为 oracle 测试实现 implementation-agnostic 的能力解耦。 前向构建流水线从源头避免 PR-derived 基准的有效性缺陷。
  • 3
    首次实证集成瓶颈。 跨模块集成——而非组件实现——是当前 LLM 仓库级生成的瓶颈。 失败分析和人类 baseline 共同确认结论稳健。 这一发现划定了 AI 辅助开发的能力边界： 组件生成可自主化，系统级集成仍需人类架构监督。
Introduction 草稿
七段式叙事结构
1
P1 — 方向重要
仓库级代码生成是新前沿：真实开发的核心挑战不是写函数，是构建多组件协作的系统。
2
P2 — 实践观察 + 诊断价值
实践中反复出现一个模式：模型生成的组件各自能通过自测，但组装成系统后出现行为错误——接口类型不匹配、状态表示不一致、错误在模块边界处被吞掉。 这意味着组件正确性和集成正确性是两种独立能力，对应不同的改进方向。 但现有基准在评估框架层面缺乏区分这两种能力的机制——既无法确认现象是否系统性存在，也无法指导能力发展方向。
3
P3 — 结构性障碍（逐个点名）
函数级/补丁级基准天然不含集成维度。仓库级基准本应是理想场所，但均存在结构性障碍： NL2RepoBench 规约已包含完整模块结构，退化为 specification-following； RepoZero 显式给出 API 签名，架构预先限定； ProgramBench 逆向设计落点在信息收集能力，且无规约支撑分层测试。 共同缺口：行为规约和分层行为测试缺一不可。
4
P4 — Spec2Repo 设计
~1000 词行为规约，只描述可观测行为，不泄露实现。模型从空目录自主构建完整仓库。 分层 oracle 测试（unit + integration）独立打分。 Integration Gap = Unit PR − Integration PR，直接量化集成能力缺口。
5
P5 — 构建方法与规模
[N] 个任务，[D] 个领域，[T] 个行为测试。 前向构建流程，所有测试 implementation-agnostic，gold solution 100% 通过。
6
P6 — 核心发现
Integration Gap 普遍且显著：最强模型 unit [X]%、integration [Y]%，gap [Z] pp。 人类 baseline gap < [H]%，证明是模型特有缺陷。 [Z]% 的集成失败归因于跨模块一致性问题。 模型能写对组件，但拼不成系统。

P7 — 贡献
