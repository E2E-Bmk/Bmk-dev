摘要

📋

我们提出 E2ECodeBench，一个专门评测大语言模型端到端完整项目交付能力的基准。给定中等信息量的产品规格文档（包含行为约束、全局不变量与制品结构），模型需要交付一个可安装、可运行的完整 Python 项目。评测采用过滤后的原始仓库测试套件作为 oracle，分为单元测试与集成测试两层，通过二者 pass rate 的差值揭示模型在跨模块一致性上的能力边界。相较于现有基准，E2ECodeBench 在任务规模、评测信号质量与对真实工程能力的对齐程度上均有显著提升。

1

背景与动机

大语言模型在代码生成领域取得了显著进展，但现有评测体系存在一个系统性盲区：几乎所有主流基准测试的都是局部代码片段或单一功能点，而非软件工程师在实际工作中面对的核心挑战——从产品需求到可交付的完整项目。

🔬 学术基准的局限

HumanEval、MBPP 等基准以函数级代码片段为单位，与真实工程场景相差甚远，无法反映模型处理系统复杂性的能力。

🏗️ 工程实践的核心挑战

真实项目要求：跨模块状态一致性、持久化与进程边界处理、错误原子性、完整的 artifact 结构、依赖管理等——这些能力至今缺乏系统评测。

随着 AI 编程助手（Cursor、GitHub Copilot 等）向"项目级自动化"演进，评测模型端到端交付完整项目的能力变得迫切。我们需要一个能够回答以下问题的基准：

❓

核心研究问题：给定足以启动项目的产品规格信息，当前最优模型能否交付一个真实可用的、完整的软件项目？其能力瓶颈在哪里？

2

问题陈述

2.1 目标能力定义

本基准测试的核心能力定义为 Definition C：端到端交付能力：

🎯

给定足以启动项目的信息（功能规格 + 行为约束 + 制品结构），模型能否交付一个真实可用的完整项目——即结构完整、行为正确、跨模块一致、可直接安装运行。

2.2 现有方法的失效模式

我们在前期工作中构建了基于 PRD 文档派生测试的评测方案，发现两个系统性失效：

❌ 失效模式一：LOC 过低

文档未指定 artifact shape → 模型默认输出单文件迷你实现 → 无法反映完整项目交付能力。

# 模型产出：kvmini.py (300行)
# 期望：src/kvstore/__init__.py
#       models.py / store.py / cli.py
❌ 失效模式二：评测信号弱

测试由模型从文档生成 → 仅覆盖文档显式描述的行为 → SOTA pass rate 过高，无法区分模型能力差异。

# oracle 信息量 ≤ 文档信息量
# 文档描述了 X%，测试只能测 X%
# SOTA 轻松通过，无区分度

相关工作与现有基准分析

基准 输入 Oracle 来源 任务粒度 测 Definition C？
NL2RepoBench 自然语言描述 + 目录结构 + API 文档（~6000词） 原始测试套件（直接使用） 完整 Python 库 部分 信息量过大，近于蓝图
PRDBench PRD 文档（~400词） 人工编写 + shell 交互测试 CLI 应用 部分 构建成本高，难以规模化
ProgramBench 编译二进制 + 文档 原始测试套件（完整） 完整程序（多语言） 接近 逆向工程而非产品开发
RepoZero 源语言完整仓库 黑盒 I/O 测试 仓库级翻译 否 翻译任务，非生成任务
SWE-bench 已有仓库 + Issue 原始测试套件 Bug 修复 / 功能补丁 否 局部修改，非完整项目
E2ECodeBench（本工作） 行为规格文档（~1000-1500词） 过滤后的原始测试套件 完整 Python 包 是 核心设计目标

关键差异化点

  • vs NL2RepoBench：文档信息量大幅减少，不提供 API 文档与代码示例，oracle 增加过滤步骤以排除文档外信息
  • vs PRDBench：测试 oracle 来自原始仓库而非人工编写，大幅降低构建成本，支持规模化
  • vs ProgramBench：输入是自然语言规格而非二进制，测试的是产品开发能力而非逆向工程能力

4.2 两个核心杠杆

杠杆一：文档信息量

解决 LOC 过低 问题。文档必须包含 artifact shape（目录结构），驱动模型产出完整包结构而非单文件实现。

  • 功能行为描述（不含实现细节）
  • 全局不变量（跨模块约束）
  • 制品结构（包名、目录树、入口文件）
  • 关键公共接口签名

杠杆二：Oracle 来源

解决 评测信号弱 问题。Oracle 来自原始仓库测试套件（过滤后），覆盖文档未明说的行为，制造模型难以逾越的能力上界。

  • 原始测试：人工编写，高质量
  • 过滤：移除与文档矛盾的测试
  • 保留：文档未禁止但未明说的行为
  • 上界：原始代码 pass rate（近 100%）

4.3 文档信息量校准原则

⚖️

判断原则：文档应包含"一位 Senior Engineer 开始项目第一天必须知道的信息"，不包含"有经验的工程师应自行决策的实现细节"。

信息类型是否包含理由
项目概述 + 类比已知系统✓ 包含激活模型训练知识，无需详述
Feature 行为描述（输入/输出/副作用）✓ 包含定义实现范围，是最基本的需求
全局不变量（跨模块约束）✓ 包含强制真实集成，防止独立 stub
制品结构（目录树、包名）✓ 包含LOC 最强杠杆，必须包含
关键公共接口签名✓ 包含Oracle 可调用的必要条件
错误行为类别✓ 包含非 happy path 是集成测试重点
内部类结构、算法选择✗ 排除工程师自行决策的实现细节
代码示例、API 使用教程✗ 排除接近实现蓝图，降低难度
测试用例、评测 checklist✗ 排除直接泄漏 oracle 信号

数据集构建方法

5.1 仓库选择标准

  • 纯 Python 包：依赖可用 venv/pip 隔离，Docker image 构建成本低
  • 有完整测试套件：原始 pytest 套件覆盖率充分，单元测试与集成测试均存在
  • 有清晰的公开文档：README 或官方文档足以提取功能规格
  • 中等复杂度：项目包含 3-6 个主要模块，存在跨模块状态交互
  • 无外部服务依赖：测试可在无网络的 Docker 容器中独立运行

5.2 文档构建流程

  1. 阅读原始仓库的 README、官方文档、源码注释
  2. 提取功能模块划分及各模块的行为描述
  3. 识别并显式化全局不变量（原始代码中隐含但文档未明说的约束）
  4. 确定 artifact shape：根据原始包结构简化为合理的目录树
  5. 提取关键公共接口签名（供 Oracle 调用的最小集合）
  6. 人工校验：Senior Engineer 能否从该文档合理开始实现？

5.3 测试套件过滤流程

对原始测试套件中的每个 test case，使用如下判断标准：

✅

保留：测试的行为与文档描述一致，或文档未提及但未禁止（文档不矛盾标准）

⚠️

人工审查：测试涉及文档边界情况，需判断是否属于合理的工程推断

❌

过滤掉：测试私有 API、内部实现细节，或与文档描述明确矛盾

5.4 Oracle 校验

过滤后的测试套件在原始代码上运行，预期 pass rate ≥ 95%。若低于此阈值，说明过滤标准需要调整（可能过滤过严或文档描述有歧义）。此 pass rate 作为评测的满分上界。

6.2 指标体系

指标计算方式测量的能力维度
Unit Pass Rate 单元测试通过数 / 单元测试总数 各功能点的独立正确性
Integration Pass Rate 集成测试通过数 / 集成测试总数 跨模块一致性与系统整体正确性
Overall Pass Rate 全部通过数 / 全部测试数（加权） 整体实现完整度
Integration Gap Unit Pass Rate − Integration Pass Rate 模型是否只实现孤立功能，无法处理系统集成
Normalized Score Overall / Oracle Pass Rate 相对于原始实现的完整度
💡

Integration Gap 是本基准的核心发现指标。我们假设当前模型在 Unit Pass Rate 较高的情况下，Integration Pass Rate 会显著下降——这一差值反映了模型在系统集成维度上的能力缺口。

6.3 Docker 评测环境

每个 task 对应一个预构建的 Docker 镜像，镜像中已安装原始仓库的所有依赖和过滤后的测试套件。评测时将模型产出 mount 到容器中执行，保证：

  • 跨 task 依赖隔离（不同项目的依赖版本不冲突）
  • 跨机器结果一致性
  • 模型产出代码的安全隔离
  • 可并发执行多个 task
7

预期贡献

🗂️ 新基准数据集

首个专注于端到端完整项目交付能力的 Python 代码生成基准，任务规模达到真实项目级别（多模块、多文件、可安装包）。

🔬 双层评测框架

单元测试 vs 集成测试的两层评测体系，通过 Integration Gap 揭示模型在系统集成维度上的能力边界，提供超越 pass rate 的诊断信息。

📐 文档设计方法论

建立"启动项目必要信息"的文档校准原则，为后续基准构建提供可复用的规格文档设计框架。

🏗️ 可扩展构建流程

测试套件过滤流程将人工构建成本降低到可接受范围，相比全手写 rubric 方案，规模化成本显著下降。

8

挑战与局限性

挑战具体表现缓解策略
过滤标准主观性 判断某测试是否"与文档矛盾"存在边界情况，不同标注者结论可能不一致 建立过滤决策树；多人独立标注取一致集；发布完整过滤记录
文档信息量校准 文档过详则难度下降；过略则模型产出混乱，无法对齐测试接口 用 baseline 模型实验验证：产出结构是否完整、接口是否可调用
接口对齐问题 模型产出的包接口与测试期望不匹配（函数名、参数不同） 文档中明确必须遵守的公共接口签名；测试 harness 增加接口检查层
任务覆盖多样性 初期任务可能集中在特定领域，泛化性待验证 扩展任务至网络库、数据处理、调度系统等多个领域
仅限 Python 当前设计针对 Python 包，不涵盖其他语言 第一版专注 Python；后续版本扩展至 Go、Rust 等
