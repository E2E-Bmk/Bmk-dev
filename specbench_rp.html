研究提案

SpecBench

仅凭行为规约，语言模型能否实现一个从未见过的代码库？

规约驱动开发是一种标准实践：团队在编写代码之前先撰写行为契约。 SpecBench 将这份契约转化为评测基准——包含 100 余个重建任务，模型只收到行为规约作为输入， 需要从零产出完整、可运行的代码库。 每个任务由一组经过验证的行为测试进行评分，这些测试仅检查可观察行为， 而非实现细节。

研究动机
开发者实际工作方式与模型评估范式之间的鸿沟

当开发者启动一个新的代码库时，典型输入是一份设计文档、一份 API 契约， 或一组行为需求——而非已有代码。 他们阅读系统必须做什么，决定内部结构，然后据此实现。 这就是规约驱动开发，它是新项目、跨语言移植和代码库重写的常态。

代码生成评测基准尚未跟上这一现实。 主流范式仍然是基于补丁：给模型一个完整的代码库，要求它应用修复。 另一条工作线给模型提供代码脚手架——函数签名、文档字符串或骨架—— 要求它填充函数体。 这两种范式的输入已经编码了大部分设计决策， 而这恰恰是开发者从规约出发时手中所没有的东西。

实际后果是存在一个测量盲区： 当前基准无法告诉我们一个模型是否能够 仅凭行为契约设计并实现一个多组件系统。 它们测量编辑能力和补全能力，但无法测量模型 做出架构选择、在多个公共接口之间维护状态一致性、 以及从文档化行为中推导出正确错误语义的能力。

技术方法
任务设计与 Oracle 构建

每个 SpecBench 任务源自一个拥有活跃测试套件和公开文档的真实 Python 库。 行为规约根据公共 API 面撰写——包括库对调用者的承诺、 在什么条件下抛出什么错误，以及跨公共接口应持有的不变量。 规约读起来像库文档，而非测试框架。 候选模型仅收到这份规约，并产出一个完整实现。

Oracle 约束一 — 行为性

任何正确的重实现都能通过，无论内部结构如何

测试经过筛选，仅检查可观察输出和副作用。 对 repr 格式、内部字段名或异常消息文本的断言被排除—— 它们反映的是某个具体实现，而非行为契约。

Oracle 约束二 — 规约可推导性

每个测试的预期结果均可追溯到规约

每个保留的测试都映射到规约的特定章节。 无法映射的测试被排除—— 失败必须反映能力差距，而非未文档化的期望。

Pipeline
How Tasks Are Constructed

Each SpecBench task is produced by a five-stage synthesis pipeline. The pipeline is deterministic in structure but iterative in execution: later stages can return feedback to earlier ones, and a task only graduates when all quality checks at every stage are satisfied.

  1. Candidate selection. A Python library is screened against hard gates: active test suite, public documentation sufficient to write a behavioral spec, no data-file or binary-heavy test dependencies that would make isolated evaluation impractical. Libraries that pass are recorded in CANDIDATES.md; those that fail are retired with a reason.
  2. Spec writing. A behavioral specification is written from the public API surface — what the library promises to callers, what errors it raises and under what conditions, and what invariants hold across its public interfaces. The spec reads as library documentation, not a benchmark artifact. Every inclusion decision is governed by two questions: does this belong to the public behavioral contract, and is it non-derivable from domain knowledge alone? Twelve validation checks must pass before the spec is considered complete.
  3. Test filtering. The library's existing test suite is audited against the oracle constraints. Tests that check internal shapes (repr format, private field names, exact exception message text) or that cannot be mapped to a spec section are excluded. Where coverage is thin, additional tests are generated from the spec — each generated test is mapped row-by-row to its source section in spec_test_map.md before it enters the oracle. The oracle must contain at least 50 tests, with no core spec section left uncovered.
  4. Candidate evaluation. The candidate model receives the behavioral specification and nothing else — no code, no tests, no prior attempts. It produces a complete multi-file implementation in isolation. The implementation is scored against the oracle at three granularities: atomic, integration, and system end-to-end. Evaluation must run on Linux or WSL; Windows runs are invalid. An import provenance check confirms the model's package is loaded from the candidate directory, not from any installed reference.
  5. Task judging. The judge audits the scoring run for cheating, solvability, and fairness before reading any score value. Failures are classified into two passes: instrument validity first (does the failing test actually check a documented, observable behavior?), then model capability (is the failure traceable to a genuine spec gap in the candidate's output?). The judge assigns one of three verdicts: QUALIFIED, BROKEN, or CHEAT_DETECTED.

The pipeline includes three feedback paths. When the judge finds a behavior present in the reference but absent from the spec, a patch request is routed back to the spec writer. When the judge finds tests that check undocumented internal shapes, a correction request is routed back to test filtering. When provenance or environment checks fail, the evaluation stage is re-run after fixing the environment. A loop that cycles more than twice without resolution retires the candidate.

Each task maintains a state machine in wip/{task}/PIPELINE_STATE.md that records the current stage, iteration counters, and history. A task reaches QUALIFIED only after all five stages complete without unresolved failures, and after a final exit checklist confirms the candidate-visible spec contains no evaluation metadata.

预期贡献
SpecBench 为社区带来什么

该基准仍在构建中——下列内容反映预期范围及当前实证发现的方向。

  • 1
    一种与实际开发实践对齐的任务定义。 规约驱动的重建隔离了现有基准所绕过的架构设计环节。 该任务更困难，也更接近开发者根据需求文档实际所做的工作。
  • 2
    从开源库构建公平重建任务的合成流水线。 该流水线通过构造产生行为性和规约可推导的 oracle 测试。 任何拥有公开文档和现有测试套件的 Python 库均可作为候选。
  • 3
    关于模型在任务各层次上能力的实证发现。 早期结果表明原子级与系统级分数之间存在显著差距—— 这是基于补丁的基准无法揭示的差距。 大多数集成和系统失败可追溯到少数几个损坏的原语（级联效应）， 而非独立的跨组件失败。 跨视图一致性和错误语义 是最频繁出现的真正能力差距。
摘要草稿
五句式结构（Farquhar 范式）
1
我们实现了什么
我们引入 SpecBench，一个包含 [N] 个 Python 库重建任务的基准， 模型仅收到行为规约作为输入，需从零产出一个完整、可运行的实现。
2
为什么困难且重要
从行为契约重建一个库需要做出架构决策、在公共接口间维护状态一致性、 并推导出正确的错误语义——这些能力是基于补丁和基于补全的基准无法探测的， 因为它们提供了规约驱动开发者必须自行产出的代码结构。
3
我们如何做到
每个任务由一个合成流水线从真实开源 Python 库构建， 该流水线根据公开 API 文档撰写行为规约，并对 oracle 测试施加两条约束进行筛选： 行为性（任何正确的重实现都能通过，无论内部结构如何） 和规约可推导性（每个测试结果均可追溯至规约标题）。
4
实证证据
在 [N] 个任务上对 [X] 个模型在三种粒度—— 原子、集成和系统端到端——进行评估，揭示了一致的能力差距： 集成级和系统级分数相对于原子级分数急剧下降， 且大多数复合失败从少数几个损坏的原语级联而来， 而非代表独立的跨组件失败。
5
最显著发现
在真正模型失败（非级联所致）中，最频繁涉及的两个能力维度 是跨视图一致性和错误语义， 即使原子级通过率超过 [Z]%，系统级通过率平均仅为 [Y]%。
