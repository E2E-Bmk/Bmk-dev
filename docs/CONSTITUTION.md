# Spec2Repo Constitution

Normative rules for every spec, oracle, and harness change. A change that
violates any MUST here is rejected regardless of who wrote it. Derived from
round-1 trajectory audit (2026-07) and the SDD/EARS requirements discipline.

## 1. Construct

The benchmark measures **system integration capability**: composing
independently-correct units into a coherent system whose cross-component and
cross-view behavior holds. Spec-to-repo reconstruction is the *instrument*,
not the construct. Consequences:

- `atomic` layer is the **control condition**. It exists to establish that
  units work (a credible premise for interpreting the gap) and to provide a
  reliable floor. It must be *completely covering* but not artificially hard.
- `integration`/`system_e2e` carry the discriminative load: cross-component
  composition, cross-view state consistency, multi-step workflows.
- Integration Gap = atomic rate − integration rate is only meaningful when
  both layers are large enough (≥15 functions each) and the atomic layer is
  positive-dominated (≥60%, target 70%).

## 2. Fairness (spec ⊇ tests)

Every scored behavior MUST be traceable to the spec: explicitly stated, or
derivable by a senior engineer from what is stated (Q2). Enforcement:

- **EARS clause discipline** — new or modified spec behavior uses one of five
  templates (ubiquitous / event-driven / state-driven / optional-feature /
  unwanted-behavior), each an atomic, testable sentence. Clause IDs live in
  `tasks/<id>/clauses.md` (auto-generated by `analysis/gen_clause_anchors.py`,
  never shipped to the candidate).
- **Traceability** — tests added or modified after this rule ships name the
  clause IDs they verify in their docstring (`Verifies: TOX-IP-001`). A test
  with no supporting clause is either checking an implementation detail
  (fix/delete the test) or has found a spec gap (patch the spec first).
- **Rule, not instance** — the spec states generating rules ("numbering starts
  at 1 and is contiguous"), never test instances ("`page_range == [1,2,3]`").
  A sentence no upstream doc would plausibly contain is test scaffolding and
  belongs in the test fix, not the spec.
- **Shared-helper audit** — every CLI flag, kwarg, env var, file-layout or
  output-format assumption inside shared test helpers/fixtures MUST be spec-
  stated or Q2-derivable. One unstated helper convention zeroes a whole layer
  (`--colored no`, `python -m pkg`, `regex_subs=`).

## 3. Environment consistency (three-way contract)

The spec's Environment section, the agent container, and the scoring container
MUST agree. The single source of truth is `oracle/<id>/requirements.txt`:

- The agent container for a task preinstalls exactly the task's oracle
  requirements (minus the target package itself) — same wheels as scoring.
- The spec's Environment section lists those packages explicitly. Never
  promise "any package on PyPI" unless the harness actually installs
  arbitrary declared dependencies at scoring time.
- The agent system prompt must not contradict the spec (no "stdlib only"
  claims when oracle tests import third-party packages).
- Provenance checks must accept every installation style that makes
  `import <pkg>` resolve to candidate code (regular, editable, `.pth`), and
  reject only genuinely foreign origins (site-packages installs of the real
  upstream package).

## 4. Test composition

Hard gates enforced by `harness/verify_task.py`:

- per layer ≥ 15 test functions; total ≥ 50
- atomic `positive` share ≥ 60% (new work targets ≥ 70%)
- zero `no_check` tests in any layer
- reference implementation passes 100% of the oracle (skips need per-test
  justification)
- dummy implementation (every public callable raises) passes 0 positive tests
- **only-add rule**: strengthening composition means adding positive tests,
  not deleting failure_path tests — error semantics are spec content

## 5. Change protocol

- Oracle or spec edits ship together with: regenerated `clauses.md`, updated
  `spec_test_map.md` (new tasks), synced `task.json` counts, and a reference
  re-run proving 0 failed / 0 error inside Docker.
- A confirmed spec_error (spec says X, reference does Y) blocks scoring until
  corrected — never recorded as a model weakness.
- Scores produced before a task-affecting fix are invalidated for that task;
  keep the ledger of which run used which task version.
